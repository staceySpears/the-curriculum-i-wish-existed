# Gate Template

Use this template to define a readiness gate for a track, project, capstone checkpoint, or major milestone.

Replace bracketed text with the specific requirements for the gate.

## Gate name

[Track or milestone name]

## Purpose

This gate verifies that the learner is ready to move from [current stage] to [next stage].

It checks not only whether the learner completed the work, but whether they can explain, debug, revise, and transfer the skill.

## Prerequisites

Before attempting this gate, the learner should have completed:

- [Required module or project]
- [Required exercises]
- [Required DSA milestone, if applicable]
- [Required reflection or review]
- [Required deployment, documentation, or benchmark, if applicable]

## Required artifact

The learner must submit:

- [Primary project or artifact]
- [Repository link or local path]
- [Tests, checks, or verification steps]
- [README or written explanation]
- [Reflection or self-evaluation]

## Correctness requirements

The artifact must meet these correctness requirements:

- [Requirement 1]
- [Requirement 2]
- [Requirement 3]

The learner must be able to show how correctness was verified.

Verification may include:

- automated tests
- manual test plan
- linting or formatting checks
- deployment smoke test
- benchmark completion
- reviewer walkthrough

## Explanation requirements

The learner must be able to explain:

- what the artifact does
- how the major pieces fit together
- what data structures, patterns, or tools were used
- why those choices were appropriate
- what tradeoffs were made
- what edge cases were considered
- what could break
- what they would improve next

## Debugging and revision requirements

The learner must provide evidence of debugging, revision, or response to feedback.

Acceptable evidence includes:

- a bug that was found and fixed
- a failing test that was made to pass
- a test added for an edge case
- a refactor with explanation
- a review comment that was addressed
- a design change based on feedback
- a deployment or environment issue that was diagnosed
- a written debugging log

## Transfer requirement

The learner must demonstrate that the skill transfers beyond the exact assignment.

Examples:

- modify the project in a small new way
- explain how the same idea applies in a different context
- solve a related problem without step-by-step instructions
- compare two possible approaches
- identify what would change at larger scale

## Self-evaluation

The learner must complete a self-evaluation before receiving a pass.

Answer the following:

1. What works well?
2. What is still fragile?
3. What was hardest?
4. What did you debug or revise?
5. What tradeoff did you make?
6. What would you improve with more time?
7. What evidence shows you are ready to move forward?

## AI disclosure

If AI materially assisted the work, describe:

- what you asked AI to help with
- what output you used
- what you changed or rejected
- how you verified the result
- what you now understand independently

If AI was not used, write: "No material AI assistance used."

## Solo learner add-ons

Solo learners should also include at least one of the following:

- recorded walkthrough
- comparison against a public example
- external review request
- expanded reflection
- extra test plan
- revisit note after 24 hours

## Team learner add-ons

Team learners should also include:

- task or issue breakdown
- pull request history
- code review evidence
- pairing or collaboration notes where relevant
- individual reflection from each teammate
- retrospective summary
- charter check or revision if needed
