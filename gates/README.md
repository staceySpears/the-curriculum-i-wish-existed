# Gates

Gates are the curriculum's readiness checks.

A gate is not a quiz, a vibe check, or a ceremonial finish line. It is the point where the learner proves they are ready to move forward.

Each track has its own gate, but all gates follow the same core idea: progress must be demonstrated through evidence.

## Why gates exist

Many learners complete assignments without becoming ready for the next level of work.

They pass tests but cannot explain the code. They finish projects but cannot debug them. They build features but cannot describe tradeoffs. They move through material but carry hidden gaps into the next stage.

Gates exist to catch that before it compounds.

A gate asks four questions:

1. Can the learner do the work?
2. Can the learner explain the work?
3. Can the learner debug or revise the work?
4. Can the learner carry the skill into a less familiar context?

If the answer to any of those questions is no, the track is not complete yet.

## What counts as evidence

Evidence depends on the track, but common forms include:

- working code
- passing tests
- a deployed application
- a project README
- a commit history
- a written reflection
- a debugging log
- a design explanation
- a code walkthrough
- a self-evaluation rubric
- a peer or mentor review
- a mock interview result
- an open source pull request
- a technical article

Evidence should be inspectable. Another person should be able to look at it and understand what the learner did, what standard was used, and why the learner is ready to continue.

## Gate components

Most gates should include five components.

### 1. Artifact

The learner produces something concrete: a program, project, repository, deployment, written packet, interview packet, or other track-specific deliverable.

### 2. Correctness check

The learner proves that the artifact behaves as expected through tests, manual verification, review, benchmark completion, or another concrete check.

### 3. Explanation

The learner explains what they built or solved, how it works, what tradeoffs they made, and what they would improve.

### 4. Revision evidence

The learner shows that they can respond to failure or feedback. This may include bug fixes, test additions, refactoring, README improvements, review responses, or a debugging log.

### 5. Reflection

The learner evaluates their own work honestly. The reflection should name strengths, weaknesses, unresolved questions, and next steps.

## Gate outcomes

A gate has three possible outcomes.

### Pass

The learner has produced enough evidence to move forward.

Passing does not mean the work is perfect. It means the learner is ready for the next level of complexity.

### Revise

The learner is close, but one or more parts need another pass.

Revision is normal. A revise outcome should be specific: what needs to change, what evidence is missing, and what standard must be met.

### Repeat

The learner is not ready to move forward and should revisit the track.

A repeat outcome does not mean the learner failed as a person. It means the foundation is not stable enough yet.

## Solo gate process

Solo learners should complete the gate even when no reviewer is available.

A solo gate should include:

- the project or artifact
- passing checks where available
- a self-evaluation rubric
- a written explanation
- a debugging or revision note
- a recorded walkthrough when possible
- an attempt to compare the work against an external standard

When no reviewer exists, the learner should read their own work as if they inherited it from someone else.

The question is:

> Would I trust this work if another person handed it to me?

## Team gate process

Team gates must evaluate both the artifact and the collaboration.

A team can ship working code while still hiding unhealthy patterns: one person doing all the work, knowledge being hoarded, conflict being avoided, or quieter teammates being pushed out.

A team gate should include:

- the shared project artifact
- tests or quality checks
- issue or task history
- pull request history
- evidence of review participation
- pairing or collaboration notes where relevant
- a retrospective
- a charter check or revision
- individual reflections from each teammate

A healthy team gate proves that the project works and that the team operated in a way that helped its members grow.

## Gate review standards

Reviewers should evaluate the work, not the learner's worth.

Good gate review is specific, evidence-based, and oriented toward readiness.

Reviewers should ask:

- Is the artifact complete enough for this stage?
- Do the tests or checks support the claim of correctness?
- Can the learner explain the important parts?
- Are the tradeoffs reasonable for this level?
- Is there evidence of debugging or revision?
- Does the learner know what is still weak?
- Would moving forward create dangerous hidden debt?

The review should end with a clear outcome: pass, revise, or repeat.

## How strict should gates be?

Gates should be humane but real.

If gates are too loose, learners carry hidden gaps forward. If gates are too harsh, learners get stuck polishing forever.

The right standard is readiness for the next track.

Not perfection.

Readiness.

## Using the template

Use `gate-template.md` when creating a gate for a track, module sequence, team milestone, or capstone checkpoint.

The template is intentionally reusable. Track-specific gates can add requirements, but they should not remove the core readiness questions.
