# Paths

This curriculum can be completed solo or with a team.

Both paths use the same technical curriculum. The difference is not what learners study. The difference is how they create feedback, accountability, review, and reflection.

A solo learner needs structure because no team is automatically watching the work.

A team learner needs structure because collaboration can hide gaps, conflict, uneven effort, and false consensus.

The path docs exist to make both modes honest.

## Choose a path

Choose the solo path if you are working mostly alone, even if you occasionally ask for review or help.

Choose the team path if you are working with the same two to four people on a recurring basis and sharing projects, review, or accountability.

You can switch paths later. If you switch, keep the same technical track sequence and adopt the operating rules for the new path.

## Solo path

The solo path emphasizes:

- self-review
- visible progress
- learning logs
- recorded walkthroughs
- external benchmarks
- deliberate requests for review
- revisiting older work

Start with `solo/README.md`.

## Team path

The team path emphasizes:

- working agreements
- shared ownership
- code review
- pairing
- retrospectives
- conflict handling
- task visibility
- knowledge sharing

Start with `team/README.md`, then read `team/happy-path.md`.

## Shared standards

Both paths require the same core evidence:

- working artifacts
- passing checks where available
- project explanations
- gate evidence
- self-evaluation
- debugging or revision evidence
- responsible AI use under `../AI_POLICY.md`

A path is not a shortcut around the curriculum. It is the operating model for completing it well.
