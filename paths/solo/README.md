# Solo Path

The solo path is for learners working mostly alone.

Solo does not mean unsupported, undisciplined, or invisible. It means you need to create the feedback loops that a team might otherwise provide.

The solo path is built around three questions:

1. What did I do?
2. What evidence proves it?
3. What should I change before moving on?

## How solo learning can go wrong

Solo learners can drift without noticing.

They can finish modules without review, skip reflection, avoid weak spots, let projects stay messy, or move forward because no one stopped them.

The solo path exists to prevent that.

## Solo operating rules

Keep a learning log.

Use project rubrics before calling work complete.

Record short walkthroughs for significant projects when possible.

Ask for outside review when the work matters.

Use gates seriously, even when no one is watching.

Revisit old work periodically and improve it.

## Required solo artifacts

A solo learner should maintain:

- a learning log
- a weekly rhythm
- project self-reviews
- gate reflections
- debugging notes for meaningful bugs
- a visible repository history
- a portfolio evidence folder or document

## Solo files

Use these docs:

- `weekly-rhythm.md`
- `accountability-plan.md`
- `self-review-rubric.md`

These are lightweight on purpose. The goal is not paperwork. The goal is honest progress.
