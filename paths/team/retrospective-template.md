# Retrospective Template

A retrospective is a short team conversation about how the work went and what should change next.

It is not a blame meeting.

It is a repair and improvement meeting.

## When to hold a retrospective

Hold a retrospective:

- at the end of each track
- after each shared project
- after a difficult conflict
- when the same blocker appears repeatedly
- before changing team agreements

## Format

Keep it simple.

### What went well?

- [Note]
- [Note]

### What was hard?

- [Note]
- [Note]

### What did we avoid?

- [Note]
- [Note]

### What should we change next time?

- [Action]
- [Action]

### Who owns each action?

- [Owner] — [Action] — [Date]

## Reflection questions

- Did everyone have meaningful work?
- Did anyone become a bottleneck?
- Did review improve the project?
- Did we communicate blockers early?
- Did we rotate learning opportunities?
- Did our charter still fit?
- What should we stop doing?
- What should we keep doing?
- What should we try next?

## Retrospective rule

Every retrospective should produce at least one small behavior change.

If nothing changes, the team is only venting.
