# Technical Communication

Technical communication is the practice of making engineering work legible to other humans.

Code matters, but code alone is rarely enough. Engineers explain decisions, write READMEs, leave commit messages, review pull requests, document APIs, describe bugs, tell project stories, and present tradeoffs.

This spine teaches learners to make their work understandable and trustworthy.

## Why this matters

A project that works but cannot be understood is hard to review, maintain, use, or hire from.

Technical communication turns work into evidence.

It helps teammates trust changes. It helps future maintainers understand intent. It helps interviewers see judgment. It helps the learner clarify their own thinking.

## The basic communication loop

### 1. Name the audience

Who needs this explanation?

A future maintainer needs different details than a user. A reviewer needs different details than an interviewer.

### 2. Name the purpose

What should the reader be able to do after reading?

Examples:

- run the project
- review the change
- understand the tradeoff
- reproduce a bug
- use an API
- evaluate a portfolio project

### 3. Give enough context

Do not assume the reader remembers what you remember.

State the problem, constraints, and relevant decisions.

### 4. Be specific

Specific writing is more useful than impressive writing.

Weak:

> Improved the app.

Better:

> Added validation for empty email fields and displayed an inline error before form submission.

## Track progression

### Track 0

Learners write notes about setup, commands, errors, and fixes.

### Track 1

Learners explain code behavior, test failures, and data structure choices in plain language.

### Track 2

Learners write change notes for existing code.

They explain what they changed, why, and what behavior was protected.

### Track 3

Learners document APIs, request and response examples, setup steps, and error behavior.

### Track 4

Learners explain user flows, accessibility choices, and browser behavior.

### Track 5

Learners document React project structure, component decisions, state flow, and deployment steps.

### Track 6

Learners translate work into interview evidence.

They write resume bullets, STAR stories, project summaries, and system design explanations.

### Track 7

Learners practice professional communication.

They write PR descriptions, review comments, technical notes, postmortems, and maintenance documentation.

### Capstone

Learners present the project clearly.

They explain the problem, users, scope, architecture, tradeoffs, tests, deployment, and next steps.

## Communication artifacts

Useful artifacts include:

- README files
- commit messages
- PR descriptions
- review comments
- API docs
- project walkthroughs
- debugging notes
- design notes
- retrospectives
- technical articles
- resume bullets
- behavioral stories

## Communication prompts

- Who is this for?
- What does the reader need to do next?
- What context would future me forget?
- What decision needs explanation?
- What tradeoff did I make?
- What changed and why?
- What evidence supports this claim?
- What should a reviewer look at first?

## Final standard

A learner practicing technical communication should make their work easier to inspect, use, review, maintain, and discuss.

The goal is not fancy writing.

The goal is useful clarity.
