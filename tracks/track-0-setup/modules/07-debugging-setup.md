# Module 07 — Debugging Setup

## Goal

Practice a calm process for setup and environment problems.

## Core idea

Setup problems are normal.

A setup problem is not proof that you cannot code. It is a sign that some assumption about files, tools, versions, paths, permissions, or commands is not true yet.

The job is to investigate.

## The setup debugging checklist

When something fails, ask:

1. What command did I run?
2. What folder was I in?
3. What exact error appeared?
4. What file or tool was the command trying to use?
5. Did this work before?
6. What changed?
7. What is the smallest thing I can test next?

## Keep a debugging note

For every meaningful setup issue, write:

- what you expected
- what happened
- what you tried
- what worked
- what you would check first next time

This note becomes evidence at the Track 0 gate.

## Practice

Intentionally create a small, safe error.

Examples:

- run a command from the wrong folder
- try to open a file using the wrong filename
- run a program before saving the file

Then use the debugging checklist to resolve it.

## Spine prompts

### Debugging

What hypothesis did you test first?

### Discipline of Doubt

What assumption caused the error?

### Technical communication

Write the fix as if explaining it to someone who is one day behind you.
