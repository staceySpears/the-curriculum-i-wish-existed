# Track 4 Gate — Frontend Fundamentals

This gate verifies that you are ready to leave Track 4 and begin building component-based applications in Track 5.

The goal is a functional, accessible UI that handles edge cases and is built on code you understand and can explain.

## Required artifact

Submit your Track 4 project.

It should include:

- a working browser application built with HTML, CSS, and JavaScript
- explicit handling of at least three edge cases: empty state, invalid input, and one other
- a README explaining what the application does and how to run it
- a self-evaluation note

## Correctness requirements

You must be able to:

- open the application in a browser and demonstrate its core functionality
- show that invalid input is handled with a visible, useful response
- show that the empty state is handled and does not break the UI
- demonstrate keyboard navigation for at least one interactive element

## Explanation requirements

You must explain:

- how user events flow from interaction to DOM update
- how JavaScript state is stored and when it changes
- why you structured the HTML the way you did
- how you made at least one element accessible
- what happens in each of your edge cases

## Debugging and revision evidence

Provide one debugging note from the track. Include:

- what you expected the browser to do
- what actually happened
- what you checked in the developer tools
- what the cause was
- what you changed

## Spine evidence

### Discipline of Doubt

Walk through your three edge cases. What assumptions did you make about user input that could have broken the UI? How did you protect against each one?

### Testing maturity

How did you verify that your application works? What did you check manually? What behavior is still unverified?

### Technical communication

Show your README. Can someone else open and use your application without asking you questions?

### Self-evaluation

What is the most fragile part of your application? What would break first under real user behavior?

### AI collaboration

Was AI used during Track 4? If so, describe one instance: what did you ask, what did it produce, how did you read and verify it before using it?

## Outcome

### Pass

You can build a functional, accessible browser application that handles edge cases and is built on code you understand.

### Revise

You are close, but must strengthen missing evidence: unhandled edge cases, missing accessibility, unexplained code, or absent debugging note.

### Repeat

You should revisit Track 4 if you cannot explain how JavaScript events and DOM updates work, cannot handle basic input validation, or cannot use the browser developer tools to investigate a problem.
